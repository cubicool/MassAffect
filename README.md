# MassAffect

A system monitoring framework for:

  - Developers who are—willingly or otherwise—forced into DevOps roles.

  - Anyone wanting a "code-first" approach to figuring out *what the hell* is
    happening to a group of agents/containers they're responsible for.

  - People who think "observability platform" shouldn't automatically mean
    "monthly bill larger than the servers you're observing."

  - Engineers who prefer writing a few lines of Python over clicking through
    twelve dashboards and three YAML files.

  - Anyone who has ever SSH'd into five machines in a row and typed `top`
    while muttering *"there has to be a better way."*

  - Developers who want monitoring that behaves more like a toolkit than a
    SaaS product.

  - People who think alerts should come from **logic they wrote**, not a
    mystery rules engine hidden behind a web UI.

  - Anyone who enjoys the phrase: *"Oh, I can just write a report for that."*

  - Developers who believe debugging production systems should involve more
    **grep and code** and less **checkboxes and drop-downs**.

  - Anyone who has ever SSH'd into a server, typed `top`, and immediately
    thought: "...this doesn't explain anything."

  - Developers who would rather write 15 lines of Python than configure
    47 dropdown menus in a monitoring UI.

  - Engineers who believe that monitoring systems should be **hackable**,
    not **clickable**.

  - People who like their infrastructure tools the same way they like
    their editors: programmable.

  - Anyone who has ever asked: "What changed?" and gotten silence from
    every dashboard.

  - Developers who want alerts driven by **logic**, not **threshold sliders**.

  - People who think Redis + Postgres + Python is a perfectly reasonable
    way to understand a distributed system.

  - Engineers who believe the best monitoring tool is the one you can
    `grep` when things go sideways.

# Not For Debate

- Code-first monitoring

- Python/Node extensibility

- Self-hosted

- Minimal UI dependency, TUI fallbacks

- No SaaS lock-in

# Overview

MassAffect consists of a central, NodeJS-based **Controller** and a variable
number of Python **Agents**. An optional **Reporter** can also be used to
amalgamate all the metrics generated by **Agents/Controllers** into
"notifications" over various backends (Slack, Mattermost, email, scripted, etc).

## Controller

Currently, the **Controller** is a very simple, single-file Express server that
waits for structured, HMAC-encrypted JSON data from *explicitly whitelisted*
**Agents** over HTTPS; both gzip-compressed and uncompressed JSON is accepted.

Once validated, the data is stored in Redis "ring buffer" list (newest events
FIRST) using the following key structure:

```
ma:agent:{AGENT_NAME}:{COLLECTOR_TYPE}:events
```

The `{COLLECTOR_TYPE}` namespace will depend on what **Agent/Collector** is
*producing* the incoming data.

### Controller (SSE View)

The **Controller** provides an SSE (Sever-Sent Events) viewing route that shows
ALL incoming JSON data from all approved sources. Only *whitelisted* IP address
are permitted.

## Agent

The **Agent** is a modern, asyncio-heavy Python 3 daemon designed to integrate
with `systemd`. It is responsible for collecting, batching, encrypting and
finally transporting (with compression) the payloads that are sent to the
**Controller**.

### Agent (Collectors)

Data payloads are gathered by *collectors*, all of which live in the `collector`
module. When launched, the **Agent** discovers and enumerates *collector*
objects; those that are marked `AUTOLOAD = True` are always initialized and run.
*Collectors* that are **not** marked `AUTOLOAD` will look for explicit
configuration variables in `config.py`.

### Agent (Socket Server/IPC)

The **Agent** also creates an "abstract" unix socket called `massaffect` which
can be used to inject data into the system that can't be easily wrapped by a
*collector*. For example, the WordPress plugin can send its output directly to
this socket, totally removing the need for an intermediate log file and
unnecessary disk IO.

As with standard *collectors*, all incoming data is validated, batched and
compressed before being queued for dispatch to the **Controller**.

### Agent (Config)

For the time being, all configuration is managed with a simple `config.py`
(which is ignored by Git, since it will contain the HMAC "secret").
